PRD Review: Website Finishing Pass — Mphasis AI & Applied Tech Lab
April 23, 2026 • 5 Advisors • 5 Peer Reviews • Chairman Synthesis

The Question

Review the PRD for the website finishing pass (unifying 9 HTML pages with 3 incompatible architectural systems):

  1. Is this the best implementation approach? What's wrong or over-engineered?
  2. What's missing that could cause problems in a multi-session AI workflow?
  3. Does it enforce page-by-page execution with approval gates and session-resumable logging?

Chairman's Verdict

Where the Council Agrees

  • The PRD is good documentation but a poor AI workflow spec. Every advisor independently arrived at the same conclusion: the 600-line document tells Claude what the website should look like but not how to verify it got there or how to resume safely. It assumes a single continuous worker with perfect memory, but the actual execution environment is stateless sessions with bounded context windows.
  • Page-by-page with approval gates is the correct model — and the PRD doesn't implement it. The current 4-phase structure cuts across pages horizontally. The user's need is vertical: finish index.html completely, get approval, then move to about.html. 4/5 peer reviews ranked The Executor strongest for identifying this mismatch.
  • Cross-page regression is the real risk. Fixing page 4 can break page 2. Copy-pasted shared elements are a "divergence machine." No advisor proposed a concrete regression detection mechanism.

Where the Council Clashes

  • How much infrastructure before touching HTML? The Outsider says "stop over-engineering, just do it." First Principles says "build constraints first or entropy wins." Resolution: at 5-9 sessions, you need lightweight verification — not a full framework, but more than nothing.
  • Is the 600-line PRD an asset or liability? It competes with file contents for AI context budget. Resolution: split into invariant spec (~100 lines, always loaded) + mutable session log (read first, written last).

Blind Spots Caught in Peer Review

  • No machine-readable canonical state. Metric numbers (25+ projects, 30+ publications, 27+ faculty) appear on multiple pages with no single-source-of-truth file. A JSON/JS constants file would eliminate an entire class of inconsistency bugs.
  • Approval gates have no verification mechanism. A new session can't confirm whether the user actually approved page 3 or the previous session just wrote "APPROVED." Needs timestamped user confirmation.
  • No re-validation before starting new work. Before touching page 5, confirm pages 1-4 are still in their approved state. Without this, regression is invisible.

The Recommendation

Restructure the PRD into three documents, then execute page-by-page starting with index.html.

Document 1: SPEC.md (~100 lines, invariant)

The canonical reference. Design tokens, font stack, nav structure, shared component patterns, metric values, content rules. Claude reads it at session start, validates against it. Never changes mid-session.

Document 2: STATUS.md (~30 lines, mutable)

Per-page status table: Page | Status | Session Date | Verified By. Statuses: NOT STARTED / IN PROGRESS / DONE-PENDING-REVIEW / APPROVED. Each APPROVED row lists shared elements present so the next session can verify nothing regressed.

Document 3: CHANGES.md (append-only log)

Every session appends what it changed, which files it touched, what it left for next time. The execution log the PRD is missing.

The One Thing to Do First

Create SPEC.md. Extract from the current PRD the exact shared elements every page must have: the <head> block, nav HTML, footer HTML, script loading pattern, and canonical metric values. Make it short enough that Claude can hold it in context alongside an entire page's HTML. Until this file exists, every session is guessing at what "consistent" means — and each will guess differently.

Advisor Responses

Contrarian Three fatal gaps: script bloat, variable drift, no execution log

The 11-script load stack is a trap. Loading GSAP, ScrollTrigger, and 6 other scripts on every page means Contact — which needs none of this — pays the full performance cost. You'll fail your own Lighthouse 90+ target on pages where these scripts do nothing.

The CSS architecture will create merge conflicts. No locked variable manifest means each session rewrites the foundation slightly differently. There's no contract saying "these 47 tokens are frozen, do not rename or remove."

No execution log with verified checksums. A new session reads "Phase 1 complete" and has no way to verify the previous session actually finished correctly. It will either redo completed work or build on a broken foundation.

The PRD describes what to build. It doesn't describe how to verify each step actually worked.

First Principles Root cause: standalone HTML is a divergence machine

The PRD answers "how to unify" without asking "why are they diverging?" Standalone HTML files with copy-pasted shared elements create drift by design. A finishing pass fixes the symptom once. The next 10 sessions recreate the problem.

What you're actually solving: How to maintain consistency in a no-build-tool environment across AI-assisted multi-session development.

The missing piece is a constraint mechanism — a machine-readable canonical reference that Claude validates every file against before editing. brand.css can drift. A validation script cannot lie.

The PRD is good engineering documentation. It's not a resilient AI workflow spec.

Expansionist This workflow itself is a publishable methodology

The PRD undersells what's here. Once unified, this becomes something genuinely rare: a university lab site that's both aesthetically credible AND factually rigorous.

The workflow — PRD-driven, AI-executed, session-logged, approval-gated — is itself publishable. A replicable methodology for small-team institutional web publishing using AI assistance.

Missing: A semantic diff log (not just "approved" but "what changed and why"). Approval gates don't check content accuracy regression, which is the lab's actual differentiator.

Biggest undervalued move: Publish the source code + PRD on GitHub as a template for other academic labs.

Outsider Over-engineered. 600 lines is too much for AI working memory.

You have 9 HTML files that need to share CSS and JS. Pick one page, make it right, copy-paste the head block to 8 others. A 600-line document introduces a new problem: the AI has to understand it while also editing code.

The AI will read the PRD, confidently say it understands, then make decisions that subtly contradict it. You'll get "I followed the PRD" alongside a <link> tag pointing to the wrong stylesheet.

Hidden flaw: If the AI breaks something on Page 4 that was already fixed on Page 2, you won't catch it until much later.

The real risk: The AI will hallucinate that it's done when it isn't. You need a mechanical checklist — literally count the <link> tags — not prose confirmation. Trust the diff, not the summary.

Executor The 4-phase structure is wrong. Here's the actual execution order.

You said page-by-page with approval gates — that's not what the PRD has. The phases cut across pages horizontally. Fix it.

Execution order: Start with index.html (most complex, sets the pattern). Then about, people, research (high-traffic). Leave contact for last.

Three gaps that will kill multi-session execution:

1. No change log. Create CHANGES.md. Every session writes: page name, what changed, what was skipped, what needs user decision.

2. No per-page "done" state. Add a status table: Page | Foundation | Structural | Components | Polish | Approved.

3. Approval mechanics undefined. Specify: user checks on localhost, replies "approved" or lists issues.

Monday action: Add status table, create CHANGES.md, instruct Claude: "Read CHANGES.md first. Update after every page. Stop and wait for approval."
